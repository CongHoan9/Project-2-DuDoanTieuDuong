GIẢI THÍCH HỆ THỐNG WEB APP DỰ ĐOÁN TIỂU ĐƯỜNG
================================================

1. TỔNG QUAN HỆ THỐNG
------------------------------------------------
Project này được chia thành 2 phần chính:

1. Frontend
- Nằm trong thư mục: frontend/
- Gồm:
  - index.html: khung giao diện
  - style.css: giao diện và bố cục
  - ui.js: các helper UI như animation số, đổi tab, tô màu trạng thái
  - app.js: logic chính của web, gọi API, render dữ liệu

2. Backend
- Nằm trong thư mục: backend/
- Dùng FastAPI để:
  - serve frontend
  - nhận request từ frontend
  - chạy model machine learning
  - lưu lịch sử vào SQLite

3. Database
- Dùng SQLite local
- File DB mặc định:
  - backend/diabetes_checks.db
- Dùng để lưu lịch sử các lần dự đoán

4. Model AI / ML
- Model đã train sẵn nằm trong:
  - backend/assets/
- Khi backend chạy, nó load các file:
  - diabetes_model.pkl
  - imputer_median.pkl
  - scaler.pkl
  - reference_stats.json


2. FRONTEND LÀM GÌ
------------------------------------------------
Frontend có nhiệm vụ:

1. Hiển thị form nhập dữ liệu
- Người dùng nhập 8 chỉ số:

  - Glucose
  - BloodPressure
  - SkinThickness
  - Insulin
  - BMI
  - DiabetesPedigreeFunction
  - Age

2. Gửi dữ liệu sang backend
- File frontend/app.js lấy dữ liệu từ form
- Sau đó gọi:
  - POST /api/predict

3. Nhận kết quả từ backend
- Backend trả về:
  - xác suất nguy cơ
  - kết luận
  - điểm nguy cơ
  - cảnh báo
  - hành động gợi ý
  - metric insights

4. Render kết quả lên giao diện
- Hiện phần kết luận chính
- Hiện risk meter
- Hiện quick signals
- Hiện radar chart
- Hiện bảng history

5. Gọi thêm các API phụ để dựng giao diện
- GET /api/reference-stats
- GET /api/model-info
- GET /api/clinical-content
- GET /api/history


3. BACKEND LÀM GÌ
------------------------------------------------
Backend là phần xử lý chính của hệ thống.

File quan trọng nhất:
- backend/main.py

Vai trò của backend:

1. Khởi tạo FastAPI app
- Tạo app API
- Gắn CORS
- include router /api

2. Tạo bảng SQLite tự động
- Ngay khi app khởi động, backend chạy:
  - Base.metadata.create_all(bind=engine)
- Mục đích:
  - nếu chưa có bảng thì tự tạo
  - không cần chạy script tạo DB riêng nữa

3. Serve frontend luôn
- Nếu thư mục frontend/ tồn tại thì backend mount:
  - /
- Kết quả:
  - chỉ cần chạy 1 web service là có cả frontend lẫn backend

4. Nhận request từ frontend
- Các route nằm trong:
  - backend/app/api/routes.py

5. Gọi prediction service
- Backend không tự viết logic model ngay trong route
- Route gọi sang:
  - backend/app/services/prediction.py

6. Lưu lịch sử vào SQLite
- Sau khi dự đoán xong, backend gọi:
  - create_check(...)
- Hàm này lưu input + output vào database SQLite


4. CÁC API ĐANG CÓ
------------------------------------------------
1. GET /api/health
- Dùng để kiểm tra backend còn sống không

4. GET /api/reference-stats

5. GET /api/model-info
- Trả về metadata của model
- Ví dụ:

6. GET /api/clinical-content
- Trả về các nội dung giải thích lâm sàng

8. QUY TRÌNH GỌI MODEL
------------------------------------------------
Model không được gọi trực tiếp từ frontend.

Quy trình đúng là:

1. Frontend gửi JSON đến:
- POST /api/predict

2. FastAPI route nhận request
- Route:
  - predict(data: PredictionInput)

3. Schema PredictionInput validate dữ liệu
- Kiểm tra:
  - có đủ field không
  - kiểu dữ liệu có hợp lệ không

4. Route gọi:
- predict_diabetes(data)

5. Trong prediction service:
- Chuyển input thành đúng thứ tự feature
- Dùng:
  - imputer_median.pkl để điền dữ liệu thiếu / zero-as-missing
  - scaler.pkl để scale dữ liệu
  - diabetes_model.pkl để predict probability

6. Sau khi model trả probability
- Service còn cộng thêm lớp logic lâm sàng:
  - tính risk score
  - phân risk band
  - sinh advice
  - sinh alerts
  - sinh metric insights
  - sinh key drivers

7. Route nhận kết quả prediction
- Sau đó gọi:
  - create_check(data.model_dump(), result)

8. create_check lưu vào SQLite
- Lưu:
  - input
  - probability
  - label
  - summary
  - timestamp

9. Route trả JSON về frontend

10. Frontend render kết quả

11. TẠI SAO FRONTEND KHÔNG GỌI THẲNG SQLITE
------------------------------------------------
Frontend chạy trong browser.
Browser không nên và gần như không thể làm việc trực tiếp với SQLite local của server theo cách an toàn.

Lý do:
- SQLite là file database trên server
- Browser chỉ nên gọi HTTP API
- Nếu cho frontend gọi thẳng DB thì:
  - sai kiến trúc
  - không an toàn
  - lộ logic hệ thống
  - không có chỗ chạy model ML

Vì vậy kiến trúc đúng là:
- Frontend -> API -> Service -> SQLite / Model

15. SEQUENCE CHI TIẾT THEO TỪNG BƯỚC
------------------------------------------------
Phần này viết theo kiểu sequence để nhìn là biết request đi qua đâu.

A. Sequence khi người dùng bấm nút Predict
------------------------------------------------
1. Người dùng bấm nút submit trên form
- File:
  - frontend/app.js
- Hàm / vị trí:
  - form.addEventListener("submit", async (event) => { ... })
- Việc xảy ra:
  - chặn reload mặc định của form bằng event.preventDefault()
  - bắt đầu luồng dự đoán

2. Frontend đọc toàn bộ dữ liệu từ form
- File:
  - frontend/app.js
- Hàm:
  - getInputPayload()
- Việc xảy ra:
  - lấy 8 input từ DOM
  - ép về int hoặc float
  - tạo object JSON để gửi sang backend

3. Frontend gọi API dự đoán
- File:
  - frontend/app.js
- Hàm:
  - buildApiUrl("predict")
  - fetch(...)
- Việc xảy ra:
  - ghép endpoint thành /api/predict
  - gửi request POST với Content-Type = application/json
  - body là JSON người dùng vừa nhập

4. Request đi vào backend route /api/predict
- File:
  - backend/app/api/routes.py
- Hàm:
  - predict(data: PredictionInput)
- Việc xảy ra:
  - FastAPI nhận request
  - tự validate request body theo schema PredictionInput
  - nếu sai field hoặc sai kiểu thì trả lỗi 422

5. Backend gọi hàm suy luận chính
- File:
  - backend/app/api/routes.py
- Hàm:
  - predict(...)
- Gọi sang:
  - backend/app/services/prediction.py
  - predict_diabetes(data)
- Việc xảy ra:
  - route chuyển phần tính toán sang service layer

6. Service chuẩn hóa input theo đúng thứ tự model cần
- File:
  - backend/app/services/prediction.py
- Hàm:
  - predict_diabetes(data)
- Việc xảy ra:
  - đọc dữ liệu theo FEATURE_ORDER
  - chuyển input thành vector số
  - xử lý các cột thuộc nhóm ZERO_AS_MISSING nếu cần

7. Service nạp các artifact phục vụ model
- File:
  - backend/app/services/prediction.py
- Các hàm cache / load liên quan:
  - các hàm load model, imputer, scaler, reference stats trong file này
- Việc xảy ra:
  - load diabetes_model.pkl
  - load imputer_median.pkl
  - load scaler.pkl
  - load reference_stats.json

8. Service chạy pipeline ML
- File:
  - backend/app/services/prediction.py
- Hàm:
  - predict_diabetes(data)
- Việc xảy ra:
  - impute median cho dữ liệu thiếu
  - scale dữ liệu
  - gọi model.predict_proba(...)
  - lấy model_probability

9. Service chạy lớp logic lâm sàng
- File:

- Hàm:
  - predict_diabetes(data)
- Việc xảy ra:

  - sinh risk_score
  - sinh risk_band
  - sinh clinical_probability
  - sinh summary
  - sinh clinical_interpretation
  - sinh advice
  - sinh alerts
  - sinh recommended_actions
  - sinh metric_insights
  - sinh key_drivers

10. Service trả object PredictionOutput về route
- File:
  - backend/app/services/prediction.py
- Hàm:
  - predict_diabetes(data)

11. Route lưu lịch sử vào SQLite

- Hàm:
  - predict(...)
- Gọi sang:
